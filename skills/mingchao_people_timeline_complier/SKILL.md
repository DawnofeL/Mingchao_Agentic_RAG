# mingchao-people-timeline-complier

核心目标：把单卷产物合并进多卷累计 JSON，保证新信息不丢、旧信息不被覆盖、跨卷引用闭环。脚本只做枚举和候选筛选，LLM 做全部阅读理解和并集写作，不靠记忆，不靠标记占位。

> ⚠️ **严禁碰源文件**：旧大 JSON 和新卷 JSON 是只读输入，全程禁止以任何形式修改、重命名、覆盖、删除这四个文件。唯一允许写出的新文件是最终合并 JSON（people + timeline 各一份）和执行过程中的 staged 中间文件。不创建任何 backup，不创建任何其他输出文件。

---

## 调用方式

用户需提供：
1. **旧大 JSON 路径** — 前几卷累计产物（people + timeline）
2. **新卷 JSON 路径** — 单卷 merger 跑完的产物（people + timeline）
3. **输出目录** — 默认与旧大 JSON 同目录

最终输出文件名取旧 JSON 起点到新卷终点的整体 chunk 范围，例如旧大 JSON 是 `chunk_1_484_*.json`、新卷是 `chunk_485_661_*.json`，最终输出为 `chunk_1_661_*.json`。

辅助脚本位于本 SKILL.md 同级的 `scripts/` 子文件夹，调用时使用绝对路径：

```bash
python $SKILL_DIR/scripts/script_find_candidates.py ...
python $SKILL_DIR/scripts/validate_kg.py ...
```

若脚本不存在，告知用户检查 skill 目录结构，**不要重新生成脚本**。

## 执行步骤

### Step 0：路径确认

确认四份 JSON 文件都存在、能正常加载、字段齐全。确认输出目录存在（不存在则创建）。

**确认后不做任何写操作**，源文件只读。

---

### Step 1：脚本候选筛选

```bash
python $SKILL_DIR/scripts/script_find_candidates.py \
    --old_people     <旧大 JSON people 路径> \
    --old_timeline   <旧大 JSON timeline 路径> \
    --new_people     <新卷 people 路径> \
    --new_timeline   <新卷 timeline 路径> \
    --output_dir     <输出目录>
```

脚本完成以下工作：

**遍历每个新人物，在旧 JSON 中查候选：**
- 命中规则：新 key 与某老 key 完全相同；新 key 出现在某老人物的 aliases 中；老 key 出现在新人物的 aliases 中；双方 aliases 交集非空；名字编辑距离/token 重叠超过阈值（兜底）
- 命中 ≥1 → 标记为"需 LLM 处理"
- 命中 0 → 标记为"直接新增"

**遍历每个新事件，在旧 JSON 中查候选：**
- 命中规则：相同 year + event 名 token 重叠超过阈值；相同 year + participants 重叠 ≥ 2 人
- 命中 ≥1 → 标记为"需 LLM 处理"
- 命中 0 → 标记为"直接新增"

**机械处理确定的部分：**
- 所有老条目原封不动复制进 `staged_people.json` / `staged_timeline.json`
- "直接新增"的新条目直接 append 进 staged 文件

**输出 `merge_workload.md`**，含所有"需 LLM 处理"的新条目及其候选老条目完整内容，格式详见"merge_workload.md 格式约定"章节。

脚本最后打印一行汇报：

```
合并 N 对候选人物 / M 对候选事件，直接新增 P 人 / Q 件，staged 文件已写出
```

---

### Step 2：LLM 处理 merge_workload

读取 `merge_workload.md`，逐条处理。每条：

1. **合并前先读 staged 文件里该老 key 的当前版本**，不能只依赖 workload.md 里的快照。原因：若同一老条目被多个新条目命中（如新卷"道衍"和"黑衣宰相"都命中老条目"姚广孝"），第一次合并后 staged 已经更新，第二次必须读最新版本，否则会覆盖第一次合并的结果。
2. **阅读新条目 + staged 里老条目的当前版本**
3. **判断同一性**（判定标准见"同一对象判定"章节）：
   - 确认是同一对象 → 按"LLM 合并守则"做字段并集，用 Edit 工具**替换** staged 文件中对应老条目
   - 确认不是同一对象 → 新条目作为独立条目 append 进 staged 文件；若 key 字面相同但确认不同人，加后缀区分（如 `张三_万历`）
   - 模棱两可 → **停下问用户**（见"停问触发清单"章节）

**遇到需要问的情形，一次性攒齐再问，不要每条都停。**

**workload 全部处理完之后，跑 validate 之前，扫一遍 staged 文件里所有 `relationships.target`、`timeline.participants`、`people.events` 字段**，把因合并或重命名产生的旧 key/旧 event 名统一替换成规范名。直接新增的条目不会自动跟着改名，这一步是修复悬空引用的唯一时机。

LLM 合并时严格遵守本文档"LLM 合并守则"章节的所有规则。

---

### Step 3：运行 validate_kg.py

```bash
python $SKILL_DIR/scripts/validate_kg.py \
    chunk_{overall_start}_{overall_end}_people.staged.json \
    chunk_{overall_start}_{overall_end}_timeline.staged.json
```

validate 包含跨卷引用闭环检查（people.events、relationships.target、timeline.participants）。**必须全部通过**，有报错就修，改完重新运行，不能跳过。

---

### Step 4：改名、清理与汇报

validate 通过后：

```bash
# staged 改为正式文件名
mv chunk_{overall_start}_{overall_end}_people.staged.json    chunk_{overall_start}_{overall_end}_people.json
mv chunk_{overall_start}_{overall_end}_timeline.staged.json  chunk_{overall_start}_{overall_end}_timeline.json

# 删除中间文件
rm merge_workload.md
```

**改名完成后，输出目录里有且仅有两个新文件**：`*_people.json` 和 `*_timeline.json`。禁止碰源文件，禁止创建 backup，禁止留下任何其他中间产物。

**向用户输出合并汇报，格式如下：**

```

============================

  chunk_{overall_start}_{overall_end}_people.json

  最终累计：Y 件
```

## LLM 合并守则

### 总体原则

1. **信息密度优先**：召回密度比简洁度重要，能保留的信息都保留，不为简洁而砍
2. **零代词**：summary、outcome、relationships.context 等所有文本字段禁止"他、她、其、此人、这个儿子、这位"等代词，必须始终用规范名字
3. **utf-8 + 无乱码**：写文件前确认中文不乱码
4. **不确定主动停下**：明确列举的触发情形必须停，模棱两可也要停，一次性攒齐再问

### 字段按合并方式分六类

| 类别 | 字段 | 合并方式 |
|---|---|---|
| A 固定枚举 | `primary_identity` / `tags[0]` | 取新值；跨大类冲突停问 |
| B 单值标量 | `year` / `era`(timeline) / `event` / `location` | 取新值或合并；明显矛盾停问 |
| C 离散列表（去近义） | `roles` / `secondary_identity` / `tags[1+]` | 字面去重 + 合并语义近义 |
| D 离散列表（不去近义） | `aliases` / `era`(people) / `events` / `source_chunks` / `participants` | 仅字面去重 |
| E 结构化列表 | `relationships` | 按 (type, target) 去重；type 语义近义合并；性质不同保留 |
| F 文本重写 | `summary` / `outcome` | LLM 阅读后重新写，取两者独立信息的并集（严禁丢失信息） |

### 类别 A：固定枚举字段

**`primary_identity`（13 枚举）**

- 双方相同 → 取共同值
- **合理迁移（自动取新值，不问）**：
  - 皇室 → 皇帝（藩王登基，如朱棣）
  - 明朝武将 → 皇帝（开国，如朱元璋）
  - 社会人员 → 文臣（弃僧入仕，如徐渭）
  - 文臣 → 皇室（追封等）
- **性质冲突（必须停问）**：
  - 武将 ↔ 宦官（如钱宁）
  - 文臣 ↔ 反叛势力
  - 宦官 ↔ 皇室
  - 外国 ↔ 任何本国身份
  - 蒙古草原 / 清 ↔ 任何本国身份
  - 其他无法归入合理迁移的差异

**`tags[0]`（7 枚举）**

- 冲突一律停问

### 类别 B：单值标量字段

**`year`**：相同取共同值；差 1 年内取新值（抄录差异）；差 >1 年停问。

**`era`(timeline)**：相同取共同值；仅写法差异统一为完整年号纪年；差很大停问。

**`event`**：取新条目的名称为统一名。

**`location`**：
- 一方是另一方的扩写（"鄱阳湖" vs "鄱阳湖水域"） → 取更完整的
- 完全不同地名 → 停问

### 类别 C：离散列表（去近义）

**字段：`roles`、`secondary_identity`、`tags[1+]`**

**可合并**（同一时期、同一职能的不同写法）：
- `["靖难首席谋士", "靖难策划者", "靖难谋主"]` → `["靖难首席谋士"]`
- `["永乐大典总编纂", "永乐大典主修"]` → `["永乐大典总编纂"]`

**不可合并**：
- 不同时期或不同职能（朱元璋 `secondary_identity = ["僧人", "红巾军将领"]` 全保留）
- 信息互补（姚广孝 `roles = ["靖难首席谋士", "永乐大典总编纂"]` 全保留）

判定核心：指向的是同一段经历吗。是 → 合；不是 → 留；不确定 → 留。

**严禁悄悄丢信息**：合并时保留信息密度更高的写法。如果两条各有独特信息、无法确认哪个更全，**保留两条，不强行合并**。"宁可冗余不要错合"在此同样适用。

### 类别 D：离散列表（不去近义）

**字段：`aliases`、`era`(people)、`events`、`source_chunks`、`participants`**

- `aliases`：召回友好，不做任何近义合并（"道衍"和"道衍和尚"都保留）
- `era`(people)：仅字面去重，按年号顺序排
- `events`：字面去重后对齐 timeline 合并后的规范 event 名
- `source_chunks`：整数去重 + 升序排
- `participants`：字面去重后对齐 people 合并后的规范 key

### 类别 E：结构化列表（relationships）

双方 relationships 全部拿出来，按 `target` 分组：

- 三元组字面完全相同 → 去重
- 同 target，type 语义指向**同一关系**（同时期同性质） → 合并到信息最具体的 type，**context 用句号连接保留所有原始信息，不重写，不精简**。context 拼接同样适用"严禁丢失信息"原则，两条都必须完整保留

  示例（可合）：朱元璋 → 陈友谅
  ```
  type="对手",     context="至正二十年至二十三年长江中游对峙"
  type="敌对",     context="应天龙湾设伏击溃陈友谅东征军"
  type="决战对手", context="鄱阳湖决战中以火攻击灭陈汉政权"
  ```
  合并为：
  ```
  type="决战对手", context="至正二十年至二十三年长江中游对峙。应天龙湾设伏击溃陈友谅东征军。鄱阳湖决战中以火攻击灭陈汉政权。"
  ```

- 同 target，type 性质不同 → 全部保留（"任命"与"处决"指向蓝玉，性质完全不同）
- 不限 relationships 总条数

**type 语义同义 vs 性质不同的硬标准**：同时期同性质 → 可合；跨时期或跨性质（"辅佐 vs 反叛"、"任命 vs 处决"）→ 不可合；不确定 → 留。

### 类别 F：文本字段（LLM 重写）

**最高优先级警告：严禁丢失任何信息。** 旧 summary 里的每一个阶段、每一件事、每一段关系，新 summary 里都必须有对应内容。"重写"是改写成连贯叙事，不是精简，不是摘要，不是截取最重要的部分。任何一卷的内容缺失都是不合格。

**`summary` 重写要求**：
- 阅读双方完整原 summary（不是片段）
- 按时间顺序重新组织，**覆盖旧 summary 和新 summary 的全部阶段、全部主要行动**
- 改写成连贯叙事，不拼接原句
- **零代词**：禁止"他、她、其、此人、这个儿子、这位"等任何代词，始终用规范名字
- **不设字数上限**，信息密度优先

## 同一对象判定

**强证据，自动合并**：
- 双方 key 完全相同
- 一方 key 出现在另一方 aliases
- 双方 aliases 有 ≥2 条交集

**弱证据，LLM 综合判**（脚本 name_similarity 兜底命中时）：
- era 完全不重叠（建文 vs 万历，相差 200 年）→ 几乎不可能同一人，分开
- era 重叠 + primary 一致 + events 有重叠 → 大概率同一人，合
- 模棱两可 → 停问

**timeline 同事件判定**：
- year 相同 + event 名 token 重叠高 + participants 主要重叠 → 同事件，event 名取更具体的
- year 相同 + participants 完全不重叠 → 不同事件
- year 差 1 年内 + 其他吻合 → 同事件，取新 year
- year 差 >1 年 → 停问

**同名不同人处理**：新条目 key 加后缀按"原 key_主要 era"格式（`张三_万历`），并同步更新所有 relationships.target 和 timeline.participants 引用。
