# mingchao-people-timeline-merger 执行逻辑

## 总体流程

```
用户提供: 一个目录路径
                    │
                    ▼
       Step 1 · 脚本提取 key，分离 passthrough / merge
       ─────────────────────────────────────────────────
       python scripts/script_extract_keys.py <目录>

       扫描所有 chunk_*_*_people.json / chunk_*_*_timeline.json
         ├─ PASSTHROUGH（仅出现 1 次）→ 直接写出
         │     passthrough_people.json
         │     passthrough_timeline.json
         └─ MERGE（出现 2+ 次）→ 写入 merge_keys.txt

       同时运行预冲突扫描（primary_identity / year / era 冲突）
         ❌ 发现冲突 → 立即停止，向用户报告，等待裁决
         ✅ 无冲突  → 继续
                    │
                    ▼
       Step 2 · 读取 merge_keys.txt，规划批次
       ─────────────────────────────────────────────────
       [PEOPLE_MERGE] 节  → 每批 3 人
       [TIMELINE_MERGE] 节 → 每批 5 件
       告知用户分批方案，但不停止工作
                    │
          ┌─────────┴─────────┐
          ▼                   ▼
   Step 3 · People 批量合并   Step 4 · Timeline 批量合并
   （每批 3 人）              （每批 5 件）
                    │
                    ▼
       Step 5 · 合成 → 排序 → 写出 → 编码自检
```

---

## Step 1 详细：script_extract_keys.py

```
脚本位置: scripts/script_extract_keys.py

输出文件（均写入目录根）：
  merge_keys.txt            ← [PEOPLE_MERGE] + [TIMELINE_MERGE] 两节
  passthrough_people.json   ← 仅出现 1 次的人物，dict 格式，按 source_chunks 升序
  passthrough_timeline.json ← 仅出现 1 次的事件，list 格式，按 year 升序

预冲突扫描内容：
  People  → 同一 key 在不同文件中 primary_identity 冲突
  Timeline → 同一 event 在不同文件中 year 或 era 冲突（仅报告实质冲突，忽略空白差异）

⚠ 扫描到冲突 → 停止，报告字段/文件/原值，请求用户裁决
✅ 无冲突    → 继续
```

## Step 3 · People 合并决策树（每批 3 人）

```
A · 提取批次数据
────────────────────────────────────────────────
python scripts/script_filter_by_key.py <目录> people <key1> [key2] [key3]

→ 写出 batch_input.json：
  {
    "type": "people",
    "keys": ["朱元璋", "汤和", "李善长"],
    "data": {
      "朱元璋": {
        "chunk_1_20_people.json": { ...record... },
        "chunk_21_40_people.json": { ...record... }
      },
      ...
    }
  }

B · LLM 合并
────────────────────────────────────────────────
读取 batch_input.json，对每个 key 按合并规则处理，
写出 batch_output.json：
  { "朱元璋": { ...merged... }, "汤和": { ...merged... }, ... }

对每个 key，逐字段决策：

  aliases / secondary_identity / era / roles / events
  → 稳定并集，去完全重复项，保留首次出现顺序

  primary_identity
  ─────────────────────────────────────────────
  两侧一致或一侧为空 → 保留有值的那侧
  两侧不同           → ⛔ 停止并报告（不自动选择）
  枚举共 13 值：皇帝/皇室/明朝武将/反叛势力/文臣/
              宦官/社会人员/蒙古草原/势力/清/朝鲜/日本/外国

  relationships
  ─────────────────────────────────────────────
  同一 target 出现多次：
    type 相同/相近 → 保留 context 最详尽的一条
    type 确实不同  → 均保留
  其余：去完全重复项，保留首次出现顺序

  source_chunks → 合并所有整数 → 去重 → 升序排序

  summary
  ─────────────────────────────────────────────
  两侧内容基本相同   → 保留较长的一条
  两侧覆盖不同时段  → [PART] 内部拼接，留待 Step 5-B 统一合成
  ⚠ 严禁在字段合并阶段改写或"美化"任何一条

C · 追加到工作 people dict（内存）

D · 清理临时文件
  rm batch_input.json  batch_output.json
```

## Step 4 · Timeline 合并决策树（每批 5 件）

```
A · 提取批次数据
────────────────────────────────────────────────
python scripts/script_filter_by_key.py <目录> timeline <event1> [event2] ...

→ batch_input.json，type = "timeline"

B · LLM 合并
────────────────────────────────────────────────
对每个 event，先做冲突检查：
  year  冲突？ → ⛔ 停止报告
  era   冲突？ → ⛔ 停止报告

无冲突后逐字段合并：

  year                      → 两侧一致保留，一侧缺失取有值，双方不同 → ⛔ 停止
  era / location / outcome  → 一致保留，一侧空取另侧，仅标点差异取排序早文件，实质冲突 → ⛔ 停止
  tags / participants       → 稳定并集，去完全重复项，保留首次出现顺序
  source_chunks             → 合并 → 去重 → 升序排序
  summary                   → 同 people summary 规则：相同取长，不同时段用 [PART] 拼接

C · 追加到工作 timeline list（内存）

D · 清理临时文件
  rm batch_input.json  batch_output.json
```

## Step 5 · 合成、排序、写出、自检

```
A · 合并 passthrough + 批次结果
────────────────────────────────────────────────
  passthrough_people.json  (dict)  +  批次合并结果 (dict)
  passthrough_timeline.json (list) +  批次合并结果 (list)

B · Summary 合成（必须执行，最终不得出现任何 [PART]）
────────────────────────────────────────────────
  扫描两文件中所有含 [PART] 的 summary
  逐条改写为统一连贯摘要：
    - 覆盖所有 [PART] 片段，不遗漏任何阶段
    - 按时间顺序组织
    - 主角 8-15 句 / 次要人物 3-5 句
    - 严禁使用代词（他/她/其等），始终用规范名字
    - 改写成连贯叙事，不拼凑原句

C · 最终排序
────────────────────────────────────────────────
  people.json:
    按 source_chunks 最小值升序
    → 首次出现顺序 → canonical key 平局

  timeline.json:
    按 year 升序
    → source_chunks 最小值升序
    → 首次出现顺序 → event 平局

D · 写出
────────────────────────────────────────────────
  json.dump(data, ensure_ascii=False, indent=2)
  文件编码: UTF-8
  禁止: shell 重定向 / PowerShell > / 字符串拼接

E · 编码自检（必须执行）
────────────────────────────────────────────────
  重新打开两个输出文件
    ├─ UTF-8 可正常解码？
    ├─ json.load() 正常解析？
    ├─ 是否含 \uFFFD 或乱码字符？
    └─ 是否含 [PART]？

      ✅ 全部通过 → 结束，告知用户输出路径
      ❌ 任一失败 → 停止并报告，严禁保留损坏文件
```

## Summary 合成：朱元璋示例

### 输入：三批文件合并后的 [PART] 拼接状态

```
chunk_1_91：   朱元璋幼名朱重八，出身钟离贫苦农家……
               1368年于应天称帝建明，最终由淮右贫民成长为
               新王朝的开创者与统一战争的主导者。
               （覆盖：早年乞讨 → 皇觉寺 → 投军 → 建明）
[PART]
chunk_92_170： 进入洪武朝中后期后，朱元璋把国家建设、
               官僚整肃和皇权重塑同时推进……朱标的突然
               去世则彻底改变了朱元璋的晚年部署。
               （覆盖：洪武制度建设 → 四大案 → 蓝玉案）
[PART]
chunk_171_263：朱元璋在本段中主要以洪武晚年的开国皇帝
               形象出现。洪武二十三年与二十九年两次命朱棣
               北征……洪武三十一年去世。
               （覆盖：朱棣北征 → 朱允炆储位 → 驾崩）
```

### ✅ 正确处理：Step 5-B 自动合成（零代词）

```
朱元璋，幼名朱重八，出身钟离贫苦农家。至正四年灾荒瘟疫并至，
父母兄侄相继饿死，朱元璋在无地葬亲的绝境中入皇觉寺为僧、流浪
讨饭，磨出极强的意志与观察力。至正十二年在汤和来信的促动下投
奔郭子兴，改名朱元璋，迎娶马秀英，并在濠州旧集团的倾轧中走向
独立。此后朱元璋连克定远、滁州，招得徐达等二十四人，得李善长
辅佐，至正十六年攻取集庆并改名应天，正式奠定争天下的核心基地。
龙湾之战以伏击重创陈友谅，后又历经鄱阳湖决战击灭陈汉、平江一
役消灭张士诚，北伐中原，于1368年正月称帝建明，年号洪武，徐达
北伐攻克大都、元顺帝出走。洪武年间，朱元璋在建设与整肃两条线
上同时发力：推行垦荒免税、恢复科举，确立八股取士制度；朝廷内
部则先借杨宪压制淮西、后纵容胡惟庸坐大，最终在洪武十三年借胡
惟庸案展开大清洗并废除丞相制度，把政务尽收于皇帝一身。此后空
印案、郭桓案、蓝玉案接连爆发，功臣宿将大量落网，锦衣卫与严刑
重典将皇权控制延伸至官僚体系每个角落。洪武晚年，朱元璋在北元
威胁与帝国继承两个方向上同步布局。洪武二十三年与二十九年两次
命朱棣北征，使朱棣的军事声望不断上升。朱标病逝后，朱元璋仍坚
持以嫡系孙支承继大统，立朱允炆为皇太孙。洪武三十一年朱元璋去
世，相信诸王与新君能共守祖制，却未意识到自己留下的恰好是一套
随时可能内爆的权力结构。
```

覆盖三批全部阶段，无 [PART]，无代词。

### ❌ 错误处理方式（只取最长那条 / 遗留 [PART]）

```
只保留 chunk_1_91 的内容，丢弃其余两批
（洪武制度建设、四大案、蓝玉案、朱棣北征全部消失）

或：字段合并后直接写出，未执行合成步骤
→ 文件中保留 [PART] 标记，无法用于 RAG 召回

错误根因：字数最多 ≠ 覆盖最全。
          各批 summary 分别写各自时段，任何一批都不完整。
          Step 5-B 合成必须在写出前完成。
```

## 失败与冲突处理

```
以下情况 → 立即停止并在对话中报告：

  ⛔ Step 1 脚本发现 primary_identity 冲突
     └─ 同一人物 key 在不同文件中 primary_identity 不同

  ⛔ Step 1 脚本发现 year / era 冲突
     └─ 同一 event 在不同文件中 year 或 era 不同（实质冲突）

  ⛔ Step 3/4 合并阶段发现人物同一性歧义
     └─ 两个不同 key 可能是同一人，但无法安全证明

  ⛔ 标量实质冲突（outcome / location 等描述性字段存在实质差异）

  ⛔ 源文件 JSON 非法或缺少必要 schema 字段
  ⛔ UTF-8 编码问题或乱码
  ⛔ 编码自检 Step 5-E 任一项不通过

报告格式（克制）：
  - 说明冲突字段、来源文件、各自原值
  - 请求用户裁决
  - 不创建任何额外的临时文件、脚本或报告
```
