# 明朝 people & timeline 合并器

## 核心原则

严格合并器（扫描、去重、排序、无损并集），输出 `mingchao_people.json` 和 `mingchao_timeline.json`。

**严禁**：改写/润色任何叙述性字段（context / outcome / location / primary_identity / roles / summary 等）；推断源 JSON 中不存在的信息；添加或删除 schema 字段；产生乱码。

**允许**：精确去重、稳定排序、无损并集、标量仅有标点/空白差异时保留排序较早文件的原值。实质冲突无法无损解决时，必须停止并报告，不得猜测或补写。

---

## 执行流程（脚本驱动）

辅助脚本位置：`<SKILL_DIR>/scripts/`（即本 SKILL.md 同级目录下的 `scripts/` 文件夹）。

```
scripts/
  script_extract_keys.py    ← Step 1：扫描所有文件，分离 passthrough / merge
  script_filter_by_key.py   ← Step 3/4：按 key 提取待合并条目，生成 batch_input.json
```

---

### Step 1 · 提取 key，分离 passthrough

```bash
python <SKILL_DIR>/scripts/script_extract_keys.py <目录路径>
```

脚本自动完成：

- 扫描目录中所有 `chunk_*_*_people.json` 和 `chunk_*_*_timeline.json`（按 chunk_start 升序，严禁按修改时间排序）
- 统计每个 key 出现在几个文件中
- **PASSTHROUGH**（仅出现 1 次）→ 直接写出到：
  - `passthrough_people.json`（dict 格式）
  - `passthrough_timeline.json`（list 格式，按 year 升序）
- **MERGE**（出现 2+ 次）→ 写入 `merge_keys.txt`
- 运行预冲突扫描：`primary_identity` 冲突、同事件 `year`/`era` 冲突

**如果脚本报告冲突 → 立即停止，向用户报告冲突详情，等待裁决，不继续任何合并步骤。**

`merge_keys.txt` 格式：
```
[PEOPLE_MERGE]
朱元璋
汤和
李善长

[TIMELINE_MERGE]
靖难之役
```

---

### Step 2 · 读取 merge_keys.txt，规划批次

读取 `merge_keys.txt`：

- `[PEOPLE_MERGE]` 节 → 每批 **3 人**
- `[TIMELINE_MERGE]` 节 → 每批 **5 件**

告知用户分批方案（如：People 9 人 → 3 批；Timeline 3 件 → 1 批）。

---

### Step 3 · 逐批合并 People

对 `[PEOPLE_MERGE]` 中每批 3 人，重复执行 A → D：

**A · 提取批次数据**

```bash
python <SKILL_DIR>/scripts/script_filter_by_key.py <目录路径> people <key1> [key2] [key3]
```

输出 `batch_input.json`（结构：`{"type":"people","keys":[...],"data":{"人名":{"文件名":record,...},...}}`）。

**B · LLM 合并**

读取 `batch_input.json`，对每个 key，按本文档的"People 合并规则"合并该 key 在各文件中的所有记录，写出 `batch_output.json`：

```json
{
  "朱元璋": { ...merged_record... },
  "汤和":   { ...merged_record... }
}
```

- 合并阶段：多批 summary 用 `\n[PART]\n` 拼接（**不要**提前合成，统一在 Step 5-B 处理）
- 严禁代词（他/她/其等），见"Summary 合成"章节

**C · 将合并结果追加到工作文件**

读取 `batch_output.json`，将结果追加进当前工作 people dict（内存中维护，最终一次性写盘）。

**D · 清理临时文件**

```bash
rm <目录路径>/batch_input.json <目录路径>/batch_output.json
```

---

### Step 4 · 逐批合并 Timeline

对 `[TIMELINE_MERGE]` 中每批 5 件，重复执行 A → D（同 Step 3，mode 换为 `timeline`）：

```bash
python <SKILL_DIR>/scripts/script_filter_by_key.py <目录路径> timeline <event1> [event2] ...
```

`batch_output.json` 格式：`{"事件名": {...merged_record...}, ...}`

---

### Step 5 · 合成、排序、写出

**A · 合并 passthrough + 批次结果**

- People：`passthrough_people.json`（dict）+ 所有批次合并结果（dict）→ 合并为完整 people dict
- Timeline：`passthrough_timeline.json`（list）+ 所有批次合并结果（list）→ 合并为完整 timeline list

**B · Summary 合成（含 [PART] 的所有 summary 必须在此步骤处理）**

见本文末"Summary 合成步骤"章节。扫描所有含 `[PART]` 的 summary，逐条改写为统一连贯摘要，**最终不得出现任何 `[PART]`**。

**C · 最终排序**

- People：按 `source_chunks` 最小值升序 → 首次出现顺序 → canonical key 平局
- Timeline：按 `year` 升序 → `source_chunks` 最小值升序 → 首次出现顺序 → `event` 平局

**D · 写出两个最终文件**

```python
json.dump(data, f, ensure_ascii=False, indent=2)  # UTF-8，禁止 shell 重定向
```

**E · 编码自检**

重新打开两个输出文件：
- UTF-8 可正常解码？
- `json.load()` 正常解析？
- 含 `\uFFFD` 或乱码字符？
- 含 `[PART]`？

✅ 全部通过 → 告知用户输出路径  
❌ 任一失败 → 修复后重写，严禁保留损坏文件

---

### 临时文件清理

Step 1 产生的 passthrough 文件和 merge_keys.txt 在 Step 5 完成后可删除，但仅限用户确认最终文件无误后。

## 编码安全

读取和写入都必须使用 UTF-8。

写文件必须满足：

- 使用 Python `open(..., encoding="utf-8")`
- 使用 `json.dump(..., ensure_ascii = False, indent = 2)`
- 严禁通过 shell 重定向写 JSON
- 严禁让 JSON 文本经过终端管道后再落盘
- 写完后必须重新用 UTF-8 打开并解析

以下情况一律视为失败：

- 输出包含 `\uFFFD`（替换字符）
- 输出无法按 UTF-8 解码
- 输出无法被 `json.load` 正常解析
- 中文出现乱码、问号乱码、形似 GBK/UTF-8 错位的 mojibake

如果出现以上任一情况，必须修复后才能结束。

## 文件发现与排序

从文件名中解析 chunk 区间，例如：

- `chunk_1_91_people.json`
- `chunk_92_170_timeline.json`

输入文件按以下规则排序：

1. `chunk_start` 升序
2. `chunk_end` 升序
3. 若仍然相同，仅为保证确定性时，`people` 排在 `timeline` 前

该排序决定融合优先级。

严禁按文件修改时间排序。

## People 合并规则

### 预期结构

每个顶层 key 是一个规范人物名，每个 value 必须保持以下固定 schema：

```json
{
  "aliases": [],
  "primary_identity": "",
  "secondary_identity": [],
  "era": [],
  "roles": [],
  "relationships": [],
  "events": [],
  "source_chunks": [],
  "summary": ""
}
```

严禁增减字段。

### 合并单位

主合并键是顶层人物 key。只有 key 完全相同，或 alias 关系可双向证明的情况下才自动合并；canonical key 取排序最早文件中的名字，严禁发明新名称或凭"看起来像"猜测合并。存在歧义时停止并报告。

### 字段规则

对 `aliases`、`secondary_identity`、`era`、`roles`、`events`：

- 做稳定并集
- 保留排序后首次出现的顺序
- 只去掉完全重复项

对 `source_chunks`：

- 合并所有整数
- 按数值升序排序
- 去掉完全重复项

对 `relationships`：

- 把每个 relationship 对象视为结构化记录
- 对**同一 target 出现多次**的情况，检查 `type` 是否相同或语义相近：
  - 若相同或相近（如多条"对手"均指向同一人）→ 保留 `context` 最详尽的一条，其余删去
  - 若确实不同（如"辅佐"与"决裂"分别指向同一人）→ 均保留
- 其余去掉完全重复项，保留首次出现顺序
- 严禁改写 `context`、`type`、`target`

对 `primary_identity`：

必须是以下 **13 个固定枚举值之一**（不能自行创造）：

| 枚举值 | 说明 |
|---|---|
| `皇帝` | 在位皇帝本人 |
| `皇室` | 太子、太后、皇后、藩王、宗室——皇室成员但非在位皇帝 |
| `武将` | 朝廷正规军将领 |
| `反叛势力` | **仅限本国**：元末割据政权（陈友谅、张士诚等）、明末农民起义（李自成、张献忠等）。后金/清作为外族政权不属于此类 |
| `文臣` | 朝廷文官与幕僚谋士 |
| `宦官` | 宦官内臣 |
| `社会人员` | 僧道、农民、商人、地主——朝廷体制外的普通人 |
| `蒙古草原` | 北元、瓦剌、鞑靼等蒙古草原政权的各类人物（含君主、将领、部落首领），如元顺帝、也先、王保保、脱古思帖木儿 |
| `势力` | 军事或政治集团本身作为条目收录，而非其中某个个人，如朵颜三卫、关宁铁骑、戚家军、东林党 |
| `清` | 女真/后金/清的各类人物（含君主、将领、使节） |
| `朝鲜` | 朝鲜半岛人物 |
| `日本` | 日本人物 |
| `外国` | 葡萄牙、荷兰等西方及其他非东亚外国人物 |

合并规则：

- 如果所有非空值完全一致，保留该值
- 如果两侧都有值且**不同**，必须停止并报告，严禁自动选择或混写
- 一侧为空、另一侧有值，取有值的一侧

不安全示例（必须报告）：

- 一侧 `"皇帝"`，另一侧 `"皇室"`
- 一侧 `"武将"`，另一侧 `"文臣"`

对 `summary`：

- **严禁取更长那条然后丢弃其余**——不同批次的 summary 往往覆盖不同时间段，"更长"不等于"更全"
- 如果两侧 summary 内容基本相同，保留较长的一条
- 如果两侧 summary 明显覆盖不同时间段（可从内容判断），先用 `\n[PART]\n` 内部拼接；**所有文件合并完成后，必须执行最终的"Summary 合成"步骤（见文末），自动将所有 [PART] 合成为统一摘要，最终输出不得含任何 `[PART]`**
- **严禁将两段 summary 直接改写成新句子来"美化"**——在合成步骤之前的字段合并阶段，任何融合都等于内容改写

## Timeline 合并规则

### 预期结构

每条 timeline 记录必须保持以下固定 schema：

```json
{
  "year": 0,
  "era": "",
  "event": "",
  "tags": [],
  "location": "",
  "participants": [],
  "outcome": "",
  "source_chunks": [],
  "summary": ""
}
```

严禁增减字段。

### 合并单位

主合并键是：

- `event` 完全相同

合并前必须额外检查：

- `year` 相同，或其中一侧缺失 `year`
- `era` 相同，或其中一侧缺失 `era`

如果两个 timeline 条目 `event` 相同，但 `year` 或 `era` 存在实质冲突，必须停止并报告。

### 字段规则

对 `tags` 和 `participants`：

- 做稳定并集
- 保留首次出现顺序
- 只去掉完全重复项

对 `source_chunks`：

- 合并所有整数
- 按数值升序排序

对 `year`：

- 保留双方一致的值
- 一边缺失、一边存在时，保留存在的一边
- 双方都存在且不同，必须停止并报告

对 `era`、`location`、`outcome`：

- 如果完全一致，保留一个
- 如果一边为空，保留另一边
- 如果差异是实质性的，严禁重写或混写成新句子
- 只有在差异明显只是标点或空白层面时，才可优先保留排序较早文件中的原值
- 否则必须停止并报告

对 `summary`，同 people summary 规则：

- 两侧内容基本相同 → 保留较长的一条
- 两侧覆盖不同时段 → 用 `\n[PART]\n` 内部拼接，留待"Summary 合成"步骤处理
- 严禁在字段合并阶段改写或"美化"

### People 综合反例：同一 `冯胜` 数据，集中所有常见错误

❌ **错误合并结果：**

```json
{
  "aliases": ["宋国公", "冯国胜", "征虏大将军"],
  "primary_identity": "明朝武将、征虏大将军、开国宿将",
  "secondary_identity": ["开国宿将"],
  "era": ["洪武"],
  "roles": ["征虏大将军", "辽东远征主帅"],
  "relationships": [
    {"type": "效忠", "target": "朱元璋", "context": "洪武年间多次北伐主帅，是朱元璋倚重的第一代武将之一。"}
  ],
  "events": ["辽东远征，纳哈出受降"],
  "source_chunks": [84, 85, 87, 91, 155, 156, 160],
  "summary": "进入洪武中期后，冯胜以征虏大将军身份主持辽东远征，与蓝玉合力逼降纳哈出，彻底瓦解辽东残元势力。晚年因功高震主及蓝玉案株连，被朱元璋赐死。"
}
```

**逐错分析：**

- `aliases` 把职衔 `"征虏大将军"` 混入别名——这是 roles 字段的值，不是人名，属于新增推断
- `aliases` 顺序错误——chunk_92_170 的 `"宋国公"` 排到了 chunk_1_91 的 `"冯国胜"` 前面，违反稳定顺序规则
- `primary_identity` 写成 `"明朝武将、征虏大将军、开国宿将"`——primary_identity 是 13 值枚举单值，严禁混写多值或混入 roles 内容
- `secondary_identity` 丢失了 chunk_1_91 的 `"红巾军将领"` 和 `"开国武将"`——早期信息被覆盖
- `era` 丢失了 `"至正"`——早期阶段被丢弃

### Timeline 综合反例：同一批数据，集中所有常见错误

❌ **场景 A 错误做法：把同年两件事合并成一条**

```json
{
  "year": 1370,
  "event": "洪武三年开科举、洪武封爵，大封功臣",
  "tags": ["政治", "科举", "封爵"],
  "outcome": "明廷一边恢复科举一边大封功臣，双线并进重建国家秩序。",
  "source_chunks": [84, 85, 94, 95]
}
```

**错误**：`event`、`tags`、`outcome` 全部被新写过，两条独立事件被压缩成一条，源真值彻底丢失。timeline 的主键是 `event`，不是 `year`；同年多件大事在历史数据里极为常见，不能按年份折叠。

---

❌ **场景 B 错误做法：year 冲突时自行选择**

假设某文件出现：
```json
{"year": 1389, "era": "洪武二十二年", "event": "捕鱼儿海之战"}
```

已有记录为 `year: 1388, era: 洪武二十一年`。

错误做法：选较早记录，直接用 1388；或取平均、或写"1388—1389年"。

**正确做法**：立即停止，报告冲突（字段 year：1388 vs 1389；era：洪武二十一年 vs 洪武二十二年；来源文件各自路径），等待用户裁决。时间锚点字段一旦实质冲突，合并器无权自行取舍。

## 最终输出排序

### `mingchao_people.json`

最终对象顺序必须确定且可复现。

按以下规则排序：

1. `source_chunks` 最小值升序
2. 再按排序后输入文件中的首次出现顺序
3. 最后才用 canonical key 做平局裁决

除非前述规则完全相同，否则不要按名字字典序硬排。

### `mingchao_timeline.json`

按以下规则排序：

1. `year` 升序
2. 再按 `source_chunks` 最小值升序
3. 再按首次出现顺序
4. 最后按 `event`

## Summary 合成步骤（所有字段合并完成后必须执行）

字段合并完成后，扫描 `mingchao_people.json` 和 `mingchao_timeline.json` 中所有包含 `[PART]` 的 summary 字段，逐条合成为统一摘要。

**合成原则：**

- 覆盖所有 `[PART]` 片段的主要内容，不遗漏任何阶段
- 按时间顺序组织
- 主角（朱元璋、朱棣等）目标 8-15 句；次要人物 3-5 句
- 改写成连贯叙事段落，消除 `[PART]` 分隔符
- 严禁只取其中一段、或拼凑原句，必须是真正改写后的统一摘要
- **严禁使用任何代词（他、她、其、这个儿子、此人等）**——必须始终用规范名字显式指代，原因是 summary 用于 RAG 召回，代词会导致语义漂移和跨文档歧义
