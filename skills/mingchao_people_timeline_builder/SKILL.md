# mingchao-kg-builder

核心目标：无论哪个会话、哪个时间点调用，产出的 JSON 结构和写作质量必须一致。一致性靠清晰的字段原则实现，而不是靠记忆上次怎么写的。

---

## 调用方式

用户需提供：
1. **chunk JSON 路径** — 等待用户指定 json 路径或者上传 json 文件
2. **chunk ID 范围** — 如 `1-91`（含两端）
3. **输出目录** — 默认与用户上传的路径相同

**默认使用 `script_chunk_extraction.py` 每批处理 20 个 chunk。** 范围超过 20 时自动拆批：

- 例如（当用户上传其他区间的 chunk_id 也使用 `script_chunk_extraction.py` 这么分）：1-91 → 1-20 / 21-40 / 41-60 / 61-80 / 81-91 （一共5批）
- 每批处理完成后立即使用 `script_incremental_merge.py` 增量写盘并做 validate，然后继续下一批

**输出命名规则：**
- 分批中间文件：`chunk_{batch_start}_{batch_end}_people.json` 和 `chunk_{batch_start}_{batch_end}_timeline.json`
- 最终汇总文件（Step 3 合成后）：`chunk_{overall_start}_{overall_end}_people.json` 和 `chunk_{overall_start}_{overall_end}_timeline.json`

---

## 执行步骤

### Step 0：确认辅助脚本已存在

辅助脚本位于**本 SKILL.md 同级的 `scripts/` 子文件夹**中。调用时使用绝对路径：

```bash
python $SKILL_DIR/scripts/script_chunk_extraction.py <chunk_json_path> <chunk_id>
python $SKILL_DIR/scripts/script_incremental_merge.py ...
python $SKILL_DIR/scripts/validate_kg.py ...
```

若脚本不存在，告知用户检查 skill 目录结构，**不要重新生成脚本**。

---

### Step 1：确定分批计划

`batch_size = 20`，例如 1-91 → `[1-20, 21-40, 41-60, 61-80, 81-91]`。告知用户分批方案后开始处理。

---

### Step 2：逐批增量循环（对每一批重复以下 A-E 步）

#### A. 逐 chunk 提取原文

对批内每个 chunk_id，依次执行：

```bash
python $SKILL_DIR/scripts/script_chunk_extraction.py <chunk_json_path> <chunk_id>
```

读取 stdout 中的原文，在**当前 LLM 上下文**中累积分析：
- 哪些人物满足收录门槛？
- 哪些事件满足收录门槛？

**注意**：收录门槛按整批（20 chunk）判断。有实质影响的事件全部收录；过渡性闲笔（如某人几点用餐且无后续影响）不收。

#### B. 输出本批 patch

分析完整批后，用 Write 工具输出两个临时文件：`patch_people.json`（只含本批新增/更新的人物）和 `patch_timeline.json`（只含本批新增/更新的事件）。**字段格式见"字段规范"章节。**

**summary 的写法（关键）：**
- **本批首次出现的人物**：写覆盖该人在本批内所有主要行动的 summary。
- **已在前批存在的人物**（如朱元璋从第1批就有）：该批 patch 的 summary **只覆盖本批时段的新内容**，不要从头重写全人生。合并脚本会将各批 summary 分段拼接保留（以 `[PART]` 为分隔符），Step 3 最终合成时才统一改写为完整摘要。

#### C. 增量合并写盘

```bash
python $SKILL_DIR/scripts/script_incremental_merge.py \
    --people_patch    patch_people.json \
    --timeline_patch  patch_timeline.json \
    --output_people   chunk_{batch_start}_{batch_end}_people.json \
    --output_timeline chunk_{batch_start}_{batch_end}_timeline.json
```

脚本输出 3 行合并报告，确认无错误后继续。

#### D. 删除临时 patch 文件

```bash
rm patch_people.json patch_timeline.json
```

#### E. 运行 validate（每批结束后必须执行）

```bash
python $SKILL_DIR/scripts/validate_kg.py \
    chunk_{batch_start}_{batch_end}_people.json \
    chunk_{batch_start}_{batch_end}_timeline.json
```

**必须全部通过才能继续下一批。** 有报错就修，改完重新运行 validate，不能跳过。validate 包含的检查见"validate 自检内容"章节。

---

### Step 3：最终合并与 summary 合成

所有批次通过 validate 后，执行最终合并步骤，输出 `chunk_{overall_start}_{overall_end}` 汇总文件。

#### 3-A. 逐批合并成最终文件

以第一批文件为初始输入，依次将后续批次合并进去：

```bash
# 初始化（直接复制第一批）
cp chunk_{b1_start}_{b1_end}_people.json   chunk_{start}_{end}_people.json
cp chunk_{b1_start}_{b1_end}_timeline.json chunk_{start}_{end}_timeline.json

# 依次合并第 2、3、... 批
python $SKILL_DIR/scripts/script_incremental_merge.py \
    --people_patch    chunk_{b2_start}_{b2_end}_people.json \
    --timeline_patch  chunk_{b2_start}_{b2_end}_timeline.json \
    --output_people   chunk_{start}_{end}_people.json \
    --output_timeline chunk_{start}_{end}_timeline.json

# （重复至最后一批）
```

#### 3-B. Summary 合成（最重要的一步）

合并脚本在增量循环中用 `[PART]` 拼接了各批的 summary 片段。现在读取最终文件，对每个包含 `[PART]` 的 summary，**根据各批片段内容合成一份统一的完整摘要**，消除 `[PART]` 分隔符。

合成原则：
- 覆盖所有批次的主要行动，不遗漏任何阶段
- 按时间顺序组织
- 主角（朱元璋、朱棣等）目标 8-15 句；次要人物 3-5 句
- 不拼接堆砌——改写成连贯的叙事段落

**示例（朱元璋，chunk_1_91 → chunk_92_170 → chunk_171_263 三批合成）：**

各批原始片段（`[PART]` 拼接后的状态）：

```
chunk_1_91：朱元璋幼名朱重八，出身钟离贫苦农家，少年长期为地主放牛，本只求有饭吃、有家可守。
至正四年灾荒瘟疫并至，父母兄侄接连饿死，他在无地葬亲的绝境中入皇觉寺为僧，又被迫云游讨饭，在
淮西与河南的流浪中磨出了极强的意志与观察世情的能力。至正十二年，他在汤和来信与官府逼迫下投奔
郭子兴，改名朱元璋，迎娶马秀英，并在濠州旧集团的倾轧中逐渐走向独立。……1368年于应天称帝建明，
旋即攻克大都、迫元顺帝北走，并在洪武初年继续以定西、应昌等战役压制北元，最终由淮右贫民成长为
新王朝的开创者与统一战争的主导者。
[PART]
chunk_92_170：进入洪武朝中后期后，朱元璋把国家建设、官僚整肃和皇权重塑同时推进。洪武三年，朱元璋推行
垦荒免税、罪徒屯田与重农抑商，并恢复科举，逐步把朱熹义理与八股写作固定为明代取士标准。……朱标
的突然去世则彻底改变了朱元璋的晚年部署。为替朱允炆扫清障碍，朱元璋发动蓝玉案并继续诛杀周德兴、傅友德、
王弼、冯胜等旧将，只留下病废的汤和。
[PART]
chunk_171_263：朱元璋在本段中主要以洪武晚年的开国皇帝形象出现。洪武二十三年与二十九年两次命朱棣
北征，使这个儿子的军事声望越来越高。朱标病逝后，他仍坚持以嫡系孙支承继大统，让朱允炆接续储位。
洪武三十一年去世时，朱元璋相信诸王与新君能够共守祖制，却没意识到自己留下的恰恰是一套随时可能
内爆的权力结构。
```

✅ **合格的合成结果（覆盖三批全部阶段，零代词）：**

> 朱元璋，幼名朱重八，出身钟离贫苦农家。至正四年灾荒瘟疫并至，父母兄侄相继饿死，朱元璋在无地葬亲的绝境中入皇觉寺为僧、流浪讨饭，磨出极强的意志与观察力。至正十二年在汤和来信的促动下投奔郭子兴，改名朱元璋，迎娶马秀英，并在濠州旧集团的倾轧中走向独立。此后朱元璋连克定远、滁州，招得徐达等二十四人，得李善长辅佐，至正十六年攻取集庆并改名应天，正式奠定争天下的核心基地。龙湾之战以伏击重创陈友谅，后又历经鄱阳湖决战击灭陈汉、平江一役消灭张士诚，北伐中原，于1368年正月称帝建明，年号洪武，徐达北伐攻克大都、元顺帝出走。洪武年间，朱元璋在建设与整肃两条线上同时发力：推行垦荒免税、恢复科举，确立八股取士制度；朝廷内部则先借杨宪压制淮西、后纵容胡惟庸坐大，最终在洪武十三年借胡惟庸案展开大清洗并废除丞相制度，把政务尽收于皇帝一身。此后空印案、郭桓案、蓝玉案接连爆发，功臣宿将大量落网，锦衣卫与严刑重典将皇权控制延伸至官僚体系每个角落。洪武晚年，朱元璋在北元威胁与帝国继承两个方向上同步布局。洪武二十三年与二十九年两次命朱棣北征，使朱棣的军事声望不断上升。朱标病逝后，朱元璋仍坚持以嫡系孙支承继大统，立朱允炆为皇太孙。洪武三十一年朱元璋去世，相信诸王与新君能共守祖制，却未意识到自己留下的恰好是一套随时可能内爆的权力结构。

❌ **不合格的错误结果（只取了 chunk_1_91 的内容）：**

> 朱元璋幼名朱重八，出身钟离贫苦农家，少年长期为地主放牛，本只求有饭吃、有家可守。至正四年灾荒瘟疫并至，父母兄侄接连饿死……最终由淮右贫民成长为新王朝的开创者与统一战争的主导者。

**为什么不合格**：chunk_92_170 的洪武朝整肃（胡惟庸案、废丞相、四大案、锦衣卫）和 chunk_171_263 的晚年布局（朱棣北征、蓝玉案、朱允炆储位）全部缺失。字数最长的那批不等于覆盖最全，必须读完所有 [PART] 片段后重新改写。

#### 3-C. Roles 压缩

增量合并后 roles 可能因各批各自写角色而超出上限。最终文件中超过 5 项的人物，根据其整体身份保留最核心的 3-5 项，删去过细的阶段性描述（如"龙湾之战总指挥"这类单次事件角色）。

#### 3-D. Relationships 去冗

对同一人物，若 relationships 中出现**同一 target 的多条条目**，检查是否是同一关系性质的重复：
- 若 `type` 相同或语义相近（如"对手"出现三次指向同一人）→ 保留 context 最详尽的一条，其余删去
- 若 `type` 确实不同（如"任命"与"决战"分别指向同一 target）→ 可保留，但 context 不应重叠

#### 3-E. 最终 validate

```bash
python $SKILL_DIR/scripts/validate_kg.py \
    chunk_{overall_start}_{overall_end}_people.json \
    chunk_{overall_start}_{overall_end}_timeline.json
```

通过后告知用户最终文件路径与人物/事件数量。

#### 3-F. 清理中间批次文件

最终 validate 通过后，删除 Step 2 阶段产生的所有分批中间文件，只保留最终的 `chunk_{overall_start}_{overall_end}_people.json` 和 `chunk_{overall_start}_{overall_end}_timeline.json`。

**边界约束（重要）：** 当某批的文件名与最终文件名完全相同时（例如输入范围本身就在 `batch_size` 之内，分批方案只有一批，`batch_start == overall_start` 且 `batch_end == overall_end`），该批文件就是最终文件，**必须跳过，不能删除**。

逐批文件依次判断后再删，例如 overall 范围 485-508、分批 `[485-504, 505-508]`：

```bash
# 最终文件为 chunk_485_508_*.json，下列两批文件名均不等于最终文件名，安全删除
rm chunk_485_504_people.json chunk_485_504_timeline.json
rm chunk_505_508_people.json chunk_505_508_timeline.json
```

反例（单批场景，overall 范围 1-20、分批 `[1-20]`）：

```bash
# chunk_1_20_*.json 既是分批文件也是最终文件，跳过，不执行 rm
```

删除完成后用 `ls chunk_*_people.json chunk_*_timeline.json` 确认盘上只剩最终那两份文件，再告知用户任务结束。

## 字段规范：people.json

每个人物严格按以下参考结构提取，字段不得增减：

```json
{
  "姚广孝": {
    "aliases": ["道衍", "黑衣宰相", "少室山人"],
    "primary_identity": "社会人员",
    "secondary_identity": ["谋士"],
    "era": ["建文", "永乐"],
    "roles": ["靖难首席谋士", "永乐大典总编纂"],
    "relationships": [
      {"type": "辅佐", "target": "朱棣", "context": "靖难之役期间献策起兵，始终坚定支持朱棣"},
      {"type": "推荐", "target": "解缙", "context": "永乐大典编纂时向朱棣推荐解缙主持"}
    ],
    "events": ["靖难之役", "永乐大典编纂"],
    "source_chunks": [185, 188, 200],
    "summary": "姚广孝，法号道衍，明初僧人出身的首席谋士，绰号黑衣宰相。靖难之役期间力劝朱棣起兵并全程参谋，是燕军胜利的核心策划者。永乐年间主持编纂《永乐大典》，以出家人身份参与最高权力却始终未还俗。"
  }
}
```

---

### 字段写法原则

`aliases`
小名、字、号、封号、庙号、谥号、绰号，所有书中实际出现过的非规范称谓全部收录。规范名本身（key）不放进 aliases。

`primary_identity`
单一值，必须从以下 **13 个固定枚举**中选最有代表性的一个，取人物在书中最终或最核心的身份：

| 枚举值 | 说明 |
|---|---|
| `皇帝` | 在位皇帝本人 |
| `皇室` | 太子、太后、皇后、藩王、宗室——皇室成员但非在位皇帝 |
| `明朝武将` | 朝廷正规军将领（包含建国前朱元璋的部将） |
| `反叛势力` | **仅限本国**：元末割据政权（陈友谅、张士诚等）、明末农民起义（李自成、张献忠等）。后金/清作为外族政权不属于此类 |
| `文臣` | 朝廷文官与幕僚谋士 |
| `宦官` | 宦官内臣 |
| `社会人员` | 僧道、农民、商人、地主——朝廷体制外的普通人 |
| `蒙古草原` | 北元、瓦剌、鞑靼等蒙古草原政权的各类人物（含君主、将领、部落首领），如元顺帝、也先、王保保、脱古思帖木儿 |
| `势力` | 军事或政治集团本身作为条目收录，而非其中某个个人，如朵颜三卫、关宁铁骑、戚家军、东林党 |
| `清` | 女真/后金/清的各类人物（含君主、将领、使节） |
| `朝鲜` | 朝鲜半岛人物 |
| `日本` | 日本人物 |
| `外国` | 葡萄牙、荷兰等西方及其他非东亚外国人物 |

**警告**：`primary_identity` 严禁自行创造，必须严格取以上 13 个之一，这是用于分类的固定枚举，不能多不能少。

示例：朱棣最终登基 → `"皇帝"`（即便他曾是藩王）。

`secondary_identity`
无固定枚举，列表，LLM 自由填写。记录历史身份转变、背景和职能补充，每项 2-6 字：

- 朱棣（primary=皇帝）：`["藩王", "明朝武将"]`
- 朱元璋（primary=皇帝）：`["僧人", "流民", "红巾军将领"]`
- 张居正（primary=文臣）：`["权臣", "改革派"]`
- 姚广孝（primary=社会人员）：`["谋士"]`

无附加身份填空列表 `[]`。

`era`
明朝年号（洪武，永乐……），不能写"明初"、"靖难时期"等描述性短语。人物活跃跨多个年号时全部列出。

`roles`
比 primary_identity 更具体的实际职能，2-4 项为宜（Step 3 合并后可压缩）。不写泛称（"谋士"是 secondary_identity 的粒度），不写历史评价（"旷世奇才"不是 role）。

`relationships`
每条必须有 `type`（关系类型名词）、`target`（规范名，不能用别名）、`context`（具体场景描述，至少 12 字，禁止写"同为文臣"等空话）。target 必须是 people.json 中已存在的 key。

`events`
从 timeline.json 中**复制粘贴**事件名，不要手写，避免拼写不一致。操作顺序：先写完 timeline，再从 timeline 复制 event 名到这里。

`source_chunks`
整数列表，只含有实质性描述（行为/关系/事件）的 chunk。被顺带提及、无具体描述的 chunk 不算。

`summary`
**宏观全量摘要，覆盖人物在书中出现的所有主要阶段和核心行动。** 只略去口角对话、细节场景（这些由原文 chunk 做 RAG）。

- 主角（朱元璋、朱棣、张居正等）可写 8-15 句
- 次要人物 3-5 句
- 严格按以下顺序：① 一句话定义身份；② 按时间顺序每个主要行动/阶段各一句；③ 关键关系与最终结局
- **禁止**：roles 字段原文照搬、"影响深远"、"意义深远"等空洞评价
- **严禁使用任何代词（他、她、其、这个儿子、此人等）**——必须始终用规范名字显式指代，原因是 summary 用于 RAG 召回，代词会导致语义漂移和跨文档歧义

**示例（朱元璋，主角级，Step 3 合成后应达到的完整版本，零代词）：**

## 字段规范：timeline.json

每个事件严格按以下参考结构提取，字段不得增减：

```json
{
  "year": 1363,
  "era": "至正二十三年",
  "event": "鄱阳湖之战",
  "tags": ["战役", "朱元璋统一战争"],
  "location": "鄱阳湖",
  "participants": ["朱元璋", "陈友谅", "常遇春"],
  "outcome": "陈友谅中箭身亡，汉政权覆灭，朱元璋基本统一南方",
  "source_chunks": [51, 52, 53],
  "summary": "1363年（至正二十三年），鄱阳湖，朱元璋与陈友谅决战，常遇春等将领参战。双方百万水军激战近两个月，朱元璋以火攻扭转劣势，陈友谅中箭身亡，汉政权瓦解。朱元璋由此消灭最强劲的对手，统一南方大局已定，此战是明朝建立的决定性前奏，此后张士诚已是强弩之末。"
}
```

### 字段写法原则

`year`
整数，公历年份。有年号纪年的一律换算，不能直接填年号。

`era`
完整年号纪年，必须写到年，如"至正二十三年"、"洪武元年"。只写"至正"不够。

`event`
历史上有专有名称的事件直接用。没有专有名称的，用"主语+动词+关键信息"格式，10-20 字。不要把叙述段落截取当事件名。

`tags`
`tags[0]` 必须是以下 **7 值类型枚举**之一：

| 类型标签 | 适用场景 |
|---|---|
| `战役` | 军事交战、攻城、围剿 |
| `起义` | 起兵、建立政权、称帝 |
| `政治` | 朝堂博弈、清洗、政变、大案、建制（含废丞相、设锦衣卫等制度变革） |
| `外交` | 外交出使、北伐、郑和下西洋等对外行动 |
| `人物节点` | 人物出生、即位、去世、流亡等个人生命节点 |
| `民生` | 经济、水利、农业、灾荒、赋税、民变 |
| `其他` | 不属于以上任何类别时使用 |

`tags[1]` 起为系列/专题标签（如"靖难之役"、"郑和下西洋"）。同一系列的重复事件加 `"序数:N"` 标签。

**警告**：`tags[0]` 严禁自行创造，必须严格取以上 7 个之一。

**明确禁止写进 tags 的内容：**
- 人名（已有 participants 字段）
- 年号/时代描述（已有 era/year 字段）

`location`
具体地名，有现代地名可加括号注释。不写"江南"、"南方"等泛地理区域。

`participants`
people.json 中的规范名（key），直接参与的主要人物，不超过 6 人。太次要没收录进 people 的参与者从这里删去，不要为了填完整而强行加人进 people。

`outcome`
1-2 句，包含具体的人、具体的结果，可从文本验证。不写"具有重大历史意义"等空话。

`summary`
**无硬性句数限制，按以下三个结构元素写，每个要素写够为止：**

- ① 时间 + 地点 + 主要参与方
- ② 事件经过核心（宏观走向，不是原文细节）
- ③ 直接结果 + 历史意义

**示例（鄱阳湖之战）：**
> 1363年（至正二十三年），鄱阳湖，朱元璋与陈友谅决战，常遇春等将领参战。双方百万水军激战近两个月，朱元璋以火攻扭转劣势，陈友谅中箭身亡，汉政权瓦解。朱元璋由此消灭最强劲的对手，统一南方大局已定，此战是明朝建立的决定性前奏，此后张士诚已是强弩之末。

**收录门槛：** 只收录有名字可命名、或是情节转折点的事件。过渡性叙述不收录。
