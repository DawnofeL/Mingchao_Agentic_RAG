# Chunk 检索题库生成器

## 参数

| 参数 | 含义 | 示例 |
|---|---|---|
| `chunk_json` | chunk JSON 绝对路径 | `/home/levizenith/SednaAI/RAG_Ming_Refine/data/raw/mingchao_chunks.json` |
| `chunk_start` | 起始 chunk_id（含） | `100` |
| `chunk_end` | 结束 chunk_id（含） | `200` |
| `n` | 每子类型题数，默认 10 | `10` |

---

## Part 1：生成题库

### Step 1：加载 chunk

严禁直接读取整个 chunk JSON（全书 1300+ 条，一次性读会越界）。唯一方式：

```bash
python3 {SKILL_DIR}/chunk_loader.py --start {chunk_start} --end {chunk_end} --json_path {chunk_json}
```

把输出完整读进来，标记每个 chunk 的 id、章节、内容。

---

### Step 2：按子类型出题

每个 sub_type 出 `n` 道，全部为单 GT：

- **description**：找对话 / 行为经过 / 细节叙述段落，问"谁 / 在哪 / 结果怎样"。
- **causation**：找因果 / 影响 / 评价段落，问"为什么 / 什么导致了 / 作者怎么评价"。

**GT 约束**：每道题 `gt_chunk_ids` 只能是 1 个。填入的 chunk_id 必须能在其 content 里直接读出完整答案，不靠拼凑。

**Keypoint 要求**：每道题必须包含 `keypoints` 字段，1-2 条，均来自该唯一 GT chunk。每条格式：

```json
{"answer": "最短自然语言答案"}
```

- `answer` 是最短能回答该问题的自然语言句子，严禁展开、严禁大段引用原文
- 写 answer 的粒度参考：问"谁"→ 直接写人名；问"为什么"→ 最短因果句；问"怎么赢"→ 最短过程句，不超过 30 字

**每道题只能有一个问句**，禁止"……，……又是……？"拼接：

禁止：`"朱元璋为龙湾之战设计了怎样的诱敌安排，陈友谅后来又是怎样进入伏击圈的？"`

允许：`"龙湾之战朱元璋这边最后是怎么赢的？"`

---

### Step 3：叙事外壳原则

出题目的是测试语义理解，不是关键词匹配。问题里一旦出现 chunk 原文的核心词汇，embedding 相似度天然极高，评测数据完全失真。

问题只能包含场景名、人物名、战役/年号/地名等章节级定位，把"做了什么 / 怎么做 / 为什么 / 结果如何"完全留给检索系统去找。

**状语粒度约束（极易违反）**：

叙事外壳里**允许**：事件名、人物名、战役/年号/地名等章节级锚点。

叙事外壳里**严禁**：场次（"最后一次""第二轮"）、具体数量（"三路""十一队"）、具体方位（"西路""左翼"）、具体地点（"军议上""大帐中"）、场景内修饰语（"在军议结束后""趁夜色"）。

**判断标准**：这个词，只有读过 chunk 原文才知道，还是读完整本书的读者都可能记得？前者禁止，后者允许。

**正反例**：

| | 问题 | 问题所在 |
|---|---|---|
| ❌ | "洪都大战前，朱文正**在最后一次军议上**给不愿留下的人什么选择？" | "最后一次军议上"是 chunk 内部场景细节 |
| ✅ | "洪都大战前，朱文正给不愿留下的人什么选择？" | 只留事件 + 人物 |
| ❌ | "洪武五年的**三路**远征里，最不被看重的**西路**最后靠哪位先锋打出连续胜绩？" | "三路"暴露数量，"西路"暴露方位 |
| ✅ | "洪武五年的远征里，最不被看重的**一路**最后靠哪位先锋打出连续胜绩？" | "一路"是模糊指代 |

### Step 4：chunk_id=51 示范

```bash
python3 {SKILL_DIR}/chunk_loader.py --start 51 --end 51 --json_path /home/levizenith/SednaAI/RAG_Ming_Refine/data/raw/mingchao_chunks.json
```

chunk 51：鄱阳湖战役开打，徐达主动请缨做先锋，率舰队突击陈友谅并取得首战胜利，原文含"先攻！徐达的猛击！"、"'我做先锋！'"、"利用自己船只的灵活性"等关键句。

**description 合格**：`"鄱阳湖战役里，朱元璋一方是谁率先进军？"` → `gt_chunk_ids: [51]`

**description 不合格**：

| 写法 | 问题 | 问题所在 |
|---|---|---|
| 关键词直连 | "徐达在鄱阳湖之战里是怎么利用船只灵活性击败陈友谅的？" | "徐达""灵活性"直接来自原文 |
| 答案泄露 | "徐达在鄱阳湖之战里是怎么把舰队分成十一队围攻敌舰的？" | "分成十一队"就是答案本身 |
| 同义替换泄露 | "鄱阳湖战役开打那会儿先锋将领用什么战术突破了陈友谅的前军？" | "先锋将领""突破前军"是核心场景的同义概括 |
| 过于模糊 | "古代水战中弱势一方怎么靠灵活性取胜？" | 泛化到任何水战，chunk 51 无法成为唯一 GT |

**causation 合格**：`"作者认为郑和舰船的巨大尺寸跟朱元璋早年哪段经历有关？"` → `gt_chunk_ids: [51]`

**causation 不合格**：

| 写法 | 问题 | 问题所在 |
|---|---|---|
| 关键词直连 | "鄱阳湖之战对明朝后来造巨舰有什么影响？" | "鄱阳湖之战""巨舰"都是原文核心词 |
| 答案泄露 | "作者说鄱阳湖之战给朱元璋留下了深刻印象，具体影响了什么？" | "给朱元璋留下了深刻印象"整句来自原文 |

### Step 5：写出 JSON

输出路径 = `dirname(chunk_json)/chunk_eval_{chunk_start}_{chunk_end}.json`

```json
[
  {
    "qna_id": "chunk_{start}-{end}_{三位数序号}",
    "sub_type": "description",
    "question": "…",
    "gt_chunk_ids": [51],
    "keypoints": [
      {"answer": "最短能回答问题的自然语言句子"}
    ]
  }
]
```

## Part 2：自检

JSON 写完后立即运行：

```bash
python3 {SKILL_DIR}/self_check.py \
    --eval_json  {output_json_path} \
    --chunk_json {chunk_json}
```

脚本输出两部分，必须全部通过才算完成：

---

### GT 配对验证

脚本逐题打印问题 + 每个 GT chunk 的前 300 字原文。逐题阅读，确认原文能直接回答问题。不能回答 → 换 GT 或重写该题，改完重新运行。

---

### 句型连续检查

脚本自动检测每个 sub_type 内是否存在连续 3 题同句型。

**description 型违规示例**（脚本会标出这三题）：

```
龙湾一役中，朱元璋设计的诱敌安排最后是怎样落到实处的？
洪都开战前，朱文正怎样分配守城任务并对将领表态？
鄱阳湖战场上，张定边直扑朱元璋后，这场危机是怎样化解的？
```

**三题全是"状语， 怎样……"，严禁重复模板！严禁！最高优先级，严禁这种模板！**

**causation 型违规示例**：

```
朱元璋为什么决定先打陈友谅？
陈友谅为什么选择先攻洪都而不是应天？
作者为什么说张士诚的格局注定了他的失败？
```

三题全是"为什么"，必须返工。

出现 `⚠️` → 修改对应题目，重新运行，直到全部 `✅`。

**可用备选句型**（必须先确认 GT chunk 能回答才能用）：

description 型——每种结构都不同，不是换个疑问词再加一个状语：
- 直接发问，无状语：`"是谁第一个跳上了采石的岸边？"`
- 数字锚定：`"朱元璋离开横涧山时带走了多少人？"`
- 书写视角：`"马姑娘是怎么给朱元璋送饭的？"`
- 人物评价：`"书里怎么评价张士诚守高邮这件事？"`

causation 型——同样要求结构各异：
- 直接归因：`"作者把陈友谅的败局根子归在哪里？"`
- 人物判断：`"刘基说服朱元璋死守应天的核心理由是什么？"`
- 结果溯源：`"郑和舰船的尺寸跟朱元璋哪段早年经历有关？"`
- 直接问因：`"脱脱修黄河为何反而加速了元朝的崩溃？"`
